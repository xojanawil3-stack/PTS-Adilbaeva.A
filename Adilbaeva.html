    1. Sapa támiyinlew usılları. Ónim hám Process sapası

    Quality Assurance (QA) — bul programmalıq támiynattı islep shıǵıw processlerin (SDLC) rejelestiriw, sistemalastırıw hám monitoring etiw arqalı juwmaqlawshı ónim sapasın kepilleytuǵın proaktiv kózqaras bolıp tabıladı. Ol qátelerdiń aldın alıwǵa qaratılǵan. Sapa eki úlken kriteriyalar toparına bólinedi:

    1. Ónim sapası (Product Quality): Juwmaqlawshı programmanıń funkcionalligi, qáwipsizligi hám ónimdarlıǵı.
    2. Process Quality (Jarayon sapası): Kod jaziw mádeniyatı, CI/CD integraciyası, analizler hám metodologiyalardın turaqlılıǵı.

    💼 Islep shıǵarıw máselesi

    Fintex joybarında hár saparı maǵlıwmatlar bazası jańalanǵanda tranzakciyalar tómenlemekte. Testerler qáteni anıqladı, biraq ol dúzetilgende, sistemanıń basqa bóliminde jańa qáte shıqtı. Process dárejesinde (QA) buni qalay dúzeysiz?

    Mashqala procestiń reaktivliginde. QA injeneri sipatında CI/CD pipeline sistemasına **Automated Integration & Regression Gates** mexanizmin qosıw kerek. Kod master tarmaqqa qosılıwınan aldın avtomatlastırılgan júklew (Performance) hám regressiv testlerden 100% tabıslı ótiwi shárt etip belgilenedi.

    2. Programmalıq támiynat sapasın qadaǵalaw basqıshları.

    Quality Control (QC) — tayar ónimniń specifikaciya hám talaplarǵa muwapıqlıǵın tekseriwshi reaktiv process. Onıń ómirlik cikli tómendegi qatań basqıshlardan ibarat: Talaplar analizi (Requirements Analysis) → Testlerdi rejelestiriw (Test Planning) → Test scenariylerin joybarlaw (Test Design) → Test ortalıǵın sazlaw (Environment Setup) → Testlerdi orinlaw (Test Execution) → Juwmaqlawshı esabat (Test Closure).

    Entry Criteria (Kiriw kriteriyaları)

    Basqıshti baslaw ushin zárúr shártler (misalı, arxitektura hújjeti tolıq hám tastıyıqlangan boliwi).

    Exit Criteria (Shıǵıw kriteriyaları)

    Basqıshti tamamlangan dep esaplaw shártleri (máselen, barlıq áhmiyetli qáteler jabılģanlıǵı).

    💼 Ámeliy Keys

    Joybar menedjeri waqıt qısqalıǵı sebepli "Test dizayn" basqıshın ótkerip jiberiwdi buyırdı. Bunıń sistemalı aqıbeti qanday boladı?

    Test-Dizaynsiz islengen testlew tártipsiz (Ad-hoc) bolip, programmanıń kútilmegen hám quramalı kombinaciyalı jolların (Edge cases) qamtıp almaydı. Nátiyjede óndiris (Production) ortalıǵında sistemalı arxitekturalıq buzılıwlar júzege keledi.

    3. Programmalıq támiynat nátiyjeliligi hám isenimliligi (MTBF, MTTR)

    Sistemanıń isenimliligi onıń belgili waqıt aralıǵında nasazlıqlarsız isley alıw qábileti bolıp tabıladı. Ol **MTBF (Mean Time Between Failures) ** hám **MTTR (Mean Time To Repair) ** mánisleri arqalı xalıq aralıq formula tiykarında esaplanadı.

    Sistemanıń Haqiyqiy Availabilty Mánisi: 99.01%
    💼 Matematikalıq másele

    Klaster serveri 1200 saat isledi hám usi waqit ishinde 3 márte avariya júz berdi. Onı tiklewge jámi 9 saat jumsaldı. Sistema turaqlılıq koefficientin esaplań.

    MTTR = 9 / 3 = 3 saat. MTBF = (1200 - 9) / 3 = 397 saat. Availability = (397 / (397 + 3)) * 100% = **99.25%**. Sistema joqarı dárejede isenimli.

    4. Makkoldiń sapa faktorı modeli.

    1977-jili jaratılgan McCall modeli programmalıq támiynat sapasın úsh tiykarģi gorizont (kózqaras) boyinsha 11 faktorga ajiratadi: Product Operation (durıslıq, isenimlilik, nátiyjelilik, qáwipsizlik, paydalanıw qolaylıǵı), Product Revision (qollap-quwatlaw ańsatlıǵı, beyimlesiwsheńlik, testlenetuģınlıq) hámProduct Transition (portativlik, qayta paydalanıwshılıq, integraciyalanıwshılıq).

    💼 Injenerlik máselesi

    Programmalıq ónim Linuxta júdá jaqsı islemekte, biraq oni Windows platformasına kóshiriwde kitapxanalar sáykes kelmedi. McCall modeline qaray qaysi faktor buzilgan?

    Bul jerde **Product Transition** toparına kiriwshi **Portability** faktori buzılgan. Mashqalanı arxitekturalıq sheshiw ushin kodti platformadan ģárezsiz etiw (misalı, Docker konteynerlerinen paydalanıw) talap etiledi.

    5. Programmalıq támiynat sapasın muǵdarlıq bahalaw (McCabe Metric)

    Tomas McCabe metodologiyası boyınsha, kodtıń logikalıq quramalılıǵı (Cyclomatic Complexity) programmadaǵı tarmaqlanıwlar sistemasın bahalaydı. Quramalılıq qanshelli joqarı bolsa, kodtı túsiniw hám testlew sonshelli qıyın boladı. Formula: $V (G) = Predicates + 1$.

    💼 Kod quramalılıǵı máselesi

    Tómendegi funkciyanıń ciklomatikalıq quramalılıǵın esaplań hám oni 100% Branch Coverage menen qaplaw ushin neshe test-keys kerekligin tabiń:

    function checkAccess(user) { if (user.isVerified && user.hasLicense) { if (user.role === "admin") { return "Full"; } else { return "Limited"; } } return "None"; }
    Kodtaǵı logikalıq shártler (predikatlar): 3 ('user.isVerified', 'user.hasLicense', 'user.role === "admin"').
    Ciklomatikalıq quramalılıq: $V (G) = 3 + 1 = 4$.
    Demek, kodtı tolıq qamtıp alıw ushın keminde **4 óz betinshe test-keys** jazılıwı shárt.

    6.ISO/IEC 25010 Sapa Kriteriyaları Standarti.

    Xalıq aralıq ISO 25010 standartı programmalıq ónim sapasın 8 tiykarģı xarakteristikaģa ajiratadı: Functional Suitability (Funkcional muwapıqlıq), Performance Efficiency (Nátiyjelilik), Compatibility (Iykemlesiwsheńlik), Usability (Qollanıw qolaylıǵı), Reliability (Isenimlilik), Security (Qáwipsizlik), Maintainability (Qollap-quwatlaw qolaylıǵı) hám Portability (Portativlik).

    Qáwipsizlik Nátiyjelilik Kodti qollap-quwatlaw sáykeslik
    💼 Standart analizi

    Sistema paydalanıwshılardıń sırlı parollerin maǵlıwmatlar bazasında ashıq (shifrlanbaǵan) halda saqlamaqta. ISO 25010 nıń qaysı sub-atribut buzılǵan?

    Bul **Security** atributiniń **Confidentiality** kishi ólshemine pútkilley qarama-qarsı. Parollar tezlik penen 'bcrypt' yamasa 'scrypt' kriptografiyalıq xeshlew algoritmleri arqalı duzlanıp (salt) shifrlanıwı kerek.

    7.Programmalıq támiynattı islep shıǵıw standartları (ISO 12207)

    ISO/IEC 12207 — programmalıq támiynattıń ómirlik cikli processlerin (Software Life Cycle Processes) tártipke salıwshı xalıqaralıq injenerlik arxitekturası standartı bolıp tabıladı. Ol proceslerdi birlemshi (Primary), shólkemlestiriwshilik (Organizational) hám qollap-quwatlawshı (Supporting) toparlarǵa ajıratadı hám joybardıń hújjetlestiriliwin qatań talap etedi.

    💼 Standartlastırıw Keysi

    Medicinalıq kardiostimulyatorlar ushın baǵdarlama jazıp atırǵan jámáátte hesh qanday texnikalıq hújjetler hám test esabatları (Test Reports) júrgizilmey atır. Bul qaysı standart buzılıwı?

    Bul jagday **ISO/IEC 12207** hám **IEEE 730 (Software Quality Assurance Plans) ** standartlarına pútkilley qayshi keledi. Insan salamatlıǵı hám qáwipsizligine baylanıslı sistemalarda hár bir basqısh hám onıń nátiyjeleri tolıq hújjetlestiriliwi shárt.

    8.Programmalıq támiynattı sertifikatlaw hám Úshinshi tárep auditi

    Sertifikatlastırıw - ónim yamasa oni islep shigariw procesleri belgili bir xalıqaralıq texnikalıq hám huqiqiy normalarga (máselen, kiberqáwipsizlik qagıydaları) muwapıqlıǵın gárezsiz, úshinshi tárep auditorlıq shólkemi tárepinen rásmiy tastıyıqlawı.

    💼 Auditorlik másele

    Bank qosımshası xalıqaralıq viza hám mastercard sistemalarına jalǵanıwı ushın qaysı májbúriy qáwipsizlik sertifikatınan ótiwi kerek?

    Bank sistemaları májbúriy túrde **PCI-DSS (Payment Card Industry Data Security Standard) ** sertifikatınan ótiwi shárt. Ol paydalanıwshılardıń karta maǵlıwmatların qayta islew, uzatıw hám saqlaw qáwipsizligin qatań audit etedi.

    9.Uliwma sapa basqarıwı (TQM) hám Deming cikli

    Total Quality Management (TQM) — bul shólkemdegi barlıq xızmetkerlerdiń qatnasıwı arqalı processler, ónimler hám xızmetlerdi turaqlı túrde jaqsılap barıwǵa tiykarlanǵan basqarıw filosofiyası bolıp esaplanadı. Onıń tiykarında Demingtiń úzliksiz aylanba esaplanǵan **PDCA (Plan-Do-Check-Act) ** cikli jatadı.

    💼 Agile-TQM Keysi

    Scrum komandası sprint juwmaǵında retro jıynalıs ótkerip qátelerdi analizledi (Check), biraq keyingi sprintte tap sol bayaǵı qáteler tákirarlandı. PDCA nıń qaysı buwını aqqan?

    Bul jerde **ACT (Háreket etiw/Standartlastırıw) basqıshı orınlanbaǵan. Anıqlanǵan kemshiliklerdi saplastırıw ushın keyingi sprint procesine jańa qatań texnikalıq sheklewler yamasa avtomatlastırılǵan tekseriw qaǵıydaları kirgiziliwi shárt edi.

    10. Óndiris procesiniń sapasın basqarıw hám Validaciya

    Sapalıqtı tekseriw eki áhmiyetli procesten ibarat:

    1. Verifikatsiya (Verification): "Ónimdi duris qurıp atırmız ba?" (Texnikalıq talaplarga, specifikaciyalarga muwapıqlıǵın tekseriw - statikalıq/dinamikalıq).
    2. Validatsiya (Validation): "Tuwri ónimdi qurıp atırmız ba?" (Ónim aqırģi paydalanıwshınıń real zárúrlikleri hám biznes maqsetlerine sáykes keliwin tekseriw).

    💼 Injenerlik máselesi

    Programma barlıq texnikalıq talaplar (SRS) tiykarında qátesiz jazıldı, biraq aqır-ayaǵında ol klient kútken biznes paydasın keltirmedi hám interfeysi paydalanıwshıǵa qolaysız dep tabıldı. Bul qaysı procestiń sácizligi?

    Bul **Validation** procesiniń sátsizligi bolip tabıladı. Ónim texnikalıq jaqtan duris qurilgan bolsa da (Verifikaciya tabıslı), bazar hám qarıydar talaplarına sáykes kelmegen. Mashqalanı saplastırıw ushın joybardıń dáslepki basqıshlarınan baslap UAT (User Acceptance Testing) scenariylerin tartıw zárúr.

    11.Texnikalıq sapa basqarıwı hám CI/CD Geytleri

    Texnikalıq sapanı basqarıw kod bazasındaǵı **Texnikalıq qarızdarlıqtı (Technical Debt) ** qadaǵalaw degeni. Buģan erisiw ushin zamanagóy DevOps konveerlerine statikalıq analizlew quralları (máselen, SonarQube) arqalı **Quality Gates** (Sapalılıq esikleri) ornatıladı.

    💼 Texnik qarız máselesi

    Programmist waqit jetispewshiligi sebepli kodti "Copy-Paste" etip tákirarladı. Sistema isledi, biraq SonarQube sapa dárwazası kodtı master tarmaqqa qosıwdı blokladı. Nege?

    Kodtıń tákirarlanıwı (Code Duplication) texnik qarzın asıradı hám programmanıń Maintainability (Qollap-quwatlanıwshılıq) koefficientin keskin túsiredi. Sapa Gate ornatilgan shek (máselen, max 3% duplikaciya) buzilgani ushin buildti toqtatqan. Kod refaktoring etiliwi shárt.

    12. CMMI (Capability Maturity Model Integration) modeli.

    CMMI modeli kompaniyanıń injenerlik hám basqarıw processleriniń jetiklik dárejesin 5 ierarxiyalıq basqıshta belgileydi:

    1-dáreje: Dáslepki 2-dáreje: basqarıladı 3-dáreje: anıqlanǵan 4-dáreje: Quantitatively Managed (Muǵdarlı bahalanatuǵın) 5-dáreje: Optimizatsiya (turaqlı rawajlanıw)
    💼 Shólkemlestiriw auditi máselesi

    Kompaniyada joybarlardıń tabısı tek ǵana bir jetekshi "juldız" programmisttiń individual sheberligine baylanıslı, hesh qanday jazılǵan processler joq. CMMI boyınsha bul qaysı dáreje?

    Bul kompaniya **CMMI Level 1 (Initial) ** dárejesinde ekenligin kórsetedi. Processler sistemalanbaǵan hám xızmetkerlerge baylanıslı bolıp, joqarı qáwipke ie. Keyingi basqıshqa ótiw ushın processlerdi hújjetlestiriw hám basqarıwdı engiziw zárúr.

    13. Shegaralıq analiz "Qara qutı" hám BVA Simulyatorı

    Shegaralıq mánis analizi (BVA) —bul ekvivalentlik klasslarınıń shetki shegaraların tekseriwge arnalgan "qara quti" testlew texnikası bolip tabıladı. Ol shegaralıq noqatlardı qatań sistema tiykarında tekseriwge járdem beredi.

    💼 Shegara analizi máselesi

    Paydalanıwshı parolı keminde 8, kóbi menen 16 belgiden ibarat bolıwı kerek. 3-noqatlı BVA texnikası boyinsha test mánislerin aniqlań.

    BVA texnikasına qaray hárbir shegara ushin 3 noqat alınadı ($Boundary - 1, Boundary, Boundary + 1$).
    Tómengi shegara (8) ushın: **7, 8, 9** uzınlıqtaǵı parollar.
    Joqarǵı shegara (16) ushın: **15, 16, 17** uzınlıqtaǵı parollar. Jámi 6 kritikalıq test noqatı payda boladı.

    14. Testlerdi rejelestiriw hám Klassikalıq V-Model

    V-Model — islep shıǵıw (shep qanat) hám testlew (oń qanat) basqıshlarınıń parallel hám simmetriyalı sáykeslik shınjırın kórsetedi. Bul modelde test scenariyları kod jazılıwınan aldın, sáykes keletuǵın dizayn basqıshlarında jobalanadı.

    💼 V-Model Arxitektura máselesi.

    Integraciyalıq test scenariyleri V-Modeldin aniq qaysi islep shigiw basqıshındaǵı texnikalıq specifikaciyalar tiykarında jazıladı?

    V-Model arxitekturasına muwapıq, **Integraciya Testleri (Integration Testing) ** joybardıń **Sistema Arxitekturası hám Joqarı dárejeli Dizayn (High-Level Design / System Architecture) ** basqıshında jaratılgan hújjetler hám moduller aralıq interfeys specifikaciyaları tiykarında joybarlastırıladı.

    15. Sertifikatlastırıwdıń huqıqıy tiykarı hám SLA Shártnamaları

    SLA (Servis dárejesi kelisimi) — bul IT-xızmet kórsetiwshi provayder hám qarıydar arasındaǵı baǵdarlamanıń islep shıǵarıwda kepillik berilgen turaqlılıq kórsetkishlerin (Uptime), óndirislik hám nasazlıqlardı saplastırıw múddetlerin qatań belgileytuǵın huqıqıy hám texnikalıq shártnama bolıp esaplanadı.

    💼 Huqıqıy SLA máselesi

    Shártnamada bulutli xizmettiń islew kepilligi (Uptime SLA) 99.9% etip belgilengen. Eger bir ayda (720 saat) sistema 2 saat dawamında pútkilley islemey qalsa (Downtime), shártnama shártleri buzılǵan esaplanadı ma?

    99.9% Uptime talabında bir ayda ruxsat etilgen maksimal toqtap qalıw waqtı (Downtime): $720 \times (1 - 0.999) = 0.72$ saattı (shama menen 43 minut) quraydı. Sistema 2 saat islemey qalģanı ushın **SLA shártleri qopal túrde buzılģan** hám provayder huqiqiy/finanslıq kompensaciya tólewge májbúr boladı.
